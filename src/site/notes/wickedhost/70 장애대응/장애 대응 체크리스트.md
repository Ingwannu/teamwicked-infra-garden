# 장애 대응 체크리스트

이 문서는 증상별로 가장 먼저 봐야 하는 확인 순서를 적어둡니다. 세부 운영 원칙은 [[wickedhost/40 운영/운영 메모|운영 메모]]를 함께 봅니다.

## 팬아웃 장애

1. [[wickedhost/20 서버/ingwprox1|ingwprox1]]에서 `10.200.1.2`, `10.200.2.2`, `10.200.3.2` 핑 확인
2. 각 백엔드에서 Oracle `10.200.x.1` 역방향 핑 확인
3. `WireGuard` 상태와 `MTU 1280` 유지 여부 확인
4. `FOU/IPIP`는 복구 테스트가 아니면 켜지 말 것
5. DNS가 `node*`, `snode*` 고정 규칙을 벗어나지 않았는지 [[wickedhost/60 DNS/Cloudflare DNS 정리|Cloudflare DNS 정리]] 확인

## WickedShip 장애

1. `https://devpanel2.teamwicked.me` 응답 확인
2. [[wickedhost/20 서버/ingwprox2|ingwprox2]]에서 `payload-app`, `mongodb`, `redis`, `devpanel2-nginx` 상태 확인
3. 이미지 업데이트 직후면 앱 내부 `3000` 리슨 여부 확인
4. ARM 이전 후 현재 운영 호스트는 [[wickedhost/20 서버/ingwprox2|ingwprox2]]라는 점을 기준으로 볼 것
5. 과거 [[wickedhost/20 서버/ingwprox4|ingwprox4]] 설정과 혼동하지 말 것

## dFlow 장애

1. `https://devpanel1.teamwicked.me` 응답 확인
2. [[wickedhost/20 서버/ingwprox3|ingwprox3]]에서 dFlow 앱/프록시 컨테이너 상태 확인
3. 같은 서버가 k3s worker도 겸하므로, 노드 문제와 앱 문제를 분리해서 볼 것

## Grafana 장애

1. `https://dashb.teamwicked.me/login` 응답 확인
2. [[wickedhost/20 서버/ingwprox1|ingwprox1]]의 `127.0.0.1:15433` PostgreSQL 브리지 리슨 확인
3. `pmacct-postgres`와 `pmacctd` 적재 상태 확인
4. collector가 old host가 아니라 로컬 DB를 보고 있는지 확인

## Pterodactyl / Blesta 장애

1. `panel.teamwicked.me` 응답 확인
2. panel 쪽은 Oracle 경로, Blesta는 [[wickedhost/20 서버/Blesta 서버|Blesta 서버]]라는 점을 분리해서 볼 것
3. node Wings API는 `401`이 살아 있음의 기준일 수 있음

## DNS / 인증서 장애

1. Cloudflare 프록시 여부 확인
2. origin 호스트가 실제 어디인지 확인
3. 인증서 SAN/CN이 도메인과 일치하는지 확인
4. 앱 이전 직후면 이전 호스트 캐시/레코드가 남아 있지 않은지 확인

## 노드 / 클러스터 장애

1. 실제 노드 조인은 UI보다 `k3s kubectl get nodes -o wide`가 기준
2. UI 미반영과 실제 k3s 조인을 혼동하지 말 것
3. Oracle 사설망 포트 reachability와 호스트 방화벽을 함께 확인
4. `worker-seoul-01`은 현재 [[wickedhost/20 서버/ingwprox3|ingwprox3]]
